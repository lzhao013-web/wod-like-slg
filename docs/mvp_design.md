# 单机类 WOD 冒险小队游戏 MVP 设计文档

## 1. 项目定位

本游戏是一款单机回合制冒险小队经营 RPG。玩家经营一支冒险小队，在有限的游戏内天数与远征次数中，通过探索副本情报、分析敌人机制、调整职业搭配、站位、装备和技能，逐步攻克原本无法通过的副本，并用获得的奖励扩展新的阵容构筑与挑战路径。

游戏不强调玩家在战斗中即时操作，而强调战前规划、战术配置、资源取舍、战报复盘和长期成长。

MVP 的核心目标不是做出大量内容，而是验证以下体验是否成立：

> 玩家能否在“发现副本问题 → 侦察/准备 → 配队/配装/设定战术 → 自动战斗 → 阅读战报 → 调整策略 → 再次挑战”的循环中获得明确的策略成就感。

---

## 2. 核心体验

### 2.1 玩家幻想

玩家不是单个英雄，而是一支冒险队伍的经营者、战术制定者和资源管理者。

玩家的主要乐趣来自：

1. 观察当前世界中刷新的副本机会。
2. 判断每个副本的威胁、收益和生命周期。
3. 决定是否侦察、挑战、暂缓或放弃。
4. 根据情报调整阵容、站位、装备、技能和撤退条件。
5. 观看自动战斗结果和战报。
6. 通过复盘发现失败原因。
7. 用获得的装备、经验、材料和新角色解锁新的挑战路径。

### 2.2 核心循环

```text
世界刷新
→ 查看副本与商店/招募/事件
→ 侦察或分析副本情报
→ 调整队伍、站位、装备、技能、战术
→ 安排远征目标
→ 结束当天
→ 自动结算远征
→ 生成战报
→ 获得奖励/承受损失
→ 日终成长与解锁
→ 进入下一天
```

### 2.3 MVP 核心判断标准

MVP 成功的标志不是“内容很多”，而是玩家在失败后能够产生类似下面的判断：

* “这个副本不是单纯数值不够，而是我缺解毒。”
* “我需要保护后排。”
* “我应该先侦察 Boss 技能再打。”
* “这次撤退条件设晚了。”
* “我可以先打矿洞做重甲，再回来挑战盗贼营地。”
* “这个副本快过期了，我要不要冒险硬打？”

## 3. MVP 范围

### 3.1 MVP 应包含的内容

MVP 应包含一个完整但小规模的 30 天游戏周期。

建议内容规模：

```text
游戏周期：30 天
每日远征次数：2 次
队伍上阵人数：最多 4 人
总角色栏位：6~8 人
职业数量：4~6 个
副本模板：8~12 个
副本实例同时存在数量：3~5 个
副本层数：3~5 层
怪物数量：15~25 种
装备数量：40~60 件
消耗品：5~8 种
词缀数量：10~15 个
最终挑战：第 30 天前后出现一个 Boss 副本
```

### 3.2 MVP 暂不包含的内容

以下内容不进入 MVP，避免系统过度膨胀：

* 大地图自由移动。
* 多城市系统。
* 联机组队。
* 拍卖行。
* 玩家手动操作战斗。
* 大量剧情分支。
* 复杂建筑经营。
* 复杂关系系统。
* 几十个职业。
* 完全程序生成副本。
* 大规模开放世界事件链。

MVP 的目标是验证“有限天数 + 副本生命周期 + 情报分层 + 自动战斗 + 战报复盘 + 阵容构筑”这套核心结构是否好玩。

---

## 4. 游戏周期与胜利目标

### 4.1 游戏周期

MVP 使用 30 天作为一局游戏的基础长度。

每天代表一个完整回合，玩家可以在当天进行规划，并在结束当天后自动结算远征。

```text
第 1 天：基础副本与新手队伍
第 5 天左右：引入副本词缀、侦察价值和异常状态
第 10 天左右：引入中级副本和明确机制克制
第 15 天左右：引入精英副本和多层资源压力
第 20 天左右：引入高级装备、职业构筑分化
第 25 天左右：最终 Boss 副本预告或解锁
第 30 天：最终挑战期限
```

### 4.2 胜利目标

MVP 的基础胜利目标：

> 在第 30 天结束前，成功通关最终 Boss 副本。

### 4.3 失败条件

MVP 可以采用软失败，而非立即 Game Over。

可能的失败条件：

1. 第 30 天结束时未能通关最终 Boss。
2. 队伍无法继续承担远征，例如主要成员全部重伤。
3. 玩家资源枯竭，无法继续恢复、修理和推进。

建议 MVP 优先使用第一种，即“时间期限失败”，因为它最契合有限天数与路线规划。

---

## 5. 每日回合流程

一个回合代表游戏内的一天。

### 5.1 第 N 天开始

系统进入新一天，显示当前天数、剩余总天数、队伍状态、可用远征次数、当前副本列表和可用经营功能。

---

### 5.2 世界刷新

世界刷新发生在每天开始时。

刷新内容包括：

1. 新副本出现。
2. 旧副本剩余天数减少。
3. 过期副本消失。
4. 商店刷新。
5. 招募列表刷新。
6. 事件刷新。
7. 部分制作、训练、治疗、修理进度推进。
8. 特定天数或条件下解锁新副本池。

副本不是永久存在的资源，而是有生命周期的机会。

每个副本实例拥有：

```text
副本模板 ID
副本实例 ID
出现天数
剩余天数
奖励次数
危险等级
层数
词缀
公开情报
侦察情报
战后复盘情报
敌人细节
特殊事件
基础奖励
额外奖励
```

---

### 5.3 规划阶段

规划阶段是玩家主要操作阶段。

玩家可以执行：

1. 查看副本列表。
2. 查看副本公开情报。
3. 派遣队员侦察副本。
4. 查看已有侦察情报。
5. 查看历史战报。
6. 调整上阵队伍。
7. 调整 3×3 阵型。
8. 更换装备。
9. 配置技能优先级。
10. 配置目标优先级。
11. 配置技能/道具使用条件。
12. 配置撤退条件。
13. 安排当天远征目标与顺序。
14. 安排训练、制作、治疗、修理等日终任务。
15. 购买商店物品。
16. 招募新队员。

MVP 中，玩家每天默认拥有 2 次远征机会。

侦察可以设计为以下两种方案之一：

方案 A：侦察占用远征次数。
方案 B：侦察占用角色行动，但不占用远征次数。

MVP 推荐使用方案 A，因为规则更简单，且能制造清晰取舍：

> 这次机会是拿来侦察，还是直接挑战？

---

### 5.4 结束当天

当玩家确认当天安排后，进入自动结算。

确认前应展示摘要：

```text
今日远征顺序：
1. 侦察：蜘蛛巢穴
2. 挑战：废弃矿洞

上阵队伍：
战士 / 牧师 / 游侠 / 盗贼

关键风险：
- 战士 HP 偏低
- 牧师解毒技能次数有限
- 盗贼武器耐久较低
```

玩家确认后，当天规划阶段结束。

---

### 5.5 讨伐结算

系统按玩家安排的顺序结算远征。

每次远征可能是：

1. 侦察副本。
2. 挑战副本。
3. 特殊事件处理。

挑战副本时，系统自动推进该副本的多层结构。

每层可能是：

1. 一场战斗。
2. 一个事件节点。
3. 一个陷阱节点。
4. 一个资源选择节点。
5. 一个 Boss 或精英战斗。

角色状态会在副本层间传递。

层间传递内容包括：

```text
HP
异常状态
部分技能次数
消耗品数量
装备耐久
撤退条件
临时增益/减益
```

讨伐结算需要处理：

1. 战斗胜利、失败或撤退。
2. 经验获取。
3. 掉落获取。
4. 消耗品扣除。
5. 技能次数扣除。
6. HP 与异常状态变化。
7. 装备耐久损耗。
8. 副本奖励次数扣除。
9. 副本是否被通关。
10. 是否解锁新内容。
11. 是否生成新情报。
12. 战报生成。

---

### 5.6 侦察结算

侦察不是纯文本按钮，而是一种有价值的战略行动。

侦察成功后，副本实例会解锁更多情报。

侦察情报可能包括：

1. 敌方阵型。
2. 主要伤害类型。
3. 敌人抗性倾向。
4. 精英怪存在。
5. 陷阱类型。
6. Boss 技能。
7. 特殊层数结构。
8. 隐藏奖励条件。
9. 推荐应对方向。
10. 撤退风险提示。

MVP 中侦察可以先做成必定成功，不引入侦察失败率。后续版本再考虑侦察职业、侦察风险和错误情报。

---

### 5.7 日终结算

日终结算发生在所有远征之后。

日终处理：

1. 制作进度推进或完成。
2. 训练进度推进或完成。
3. 治疗进度推进。
4. 装备修理完成。
5. 角色升级。
6. 新技能解锁。
7. 新副本池解锁。
8. 特定事件触发。
9. 进入第 N+1 天。

---

## 6. 队伍与角色系统

### 6.1 队伍规模

MVP 推荐：

```text
上阵人数：最多 4 人
总角色栏位：6~8 人
```

这样玩家有替补和轮换空间，但不会管理负担过高。

### 6.2 角色核心属性

角色拥有以下基础属性：

```text
HP：生命值
攻击：影响基础伤害
防御：减少物理伤害
速度：影响行动顺序和行动频率
命中：影响攻击命中率
闪避：影响被命中概率
抗性：减少特定类型伤害或异常状态
行动次数：每回合可行动次数或行动点
```

MVP 中抗性可以先简化为：

```text
物理抗性
毒抗性
魔法抗性
诅咒抗性
火焰抗性
```

不需要一开始做过多伤害类型。

### 6.3 职业设计

MVP 推荐 4 个基础职业，最多扩展到 6 个。

基础职业：

#### 战士

定位：前排承伤、保护队友、稳定物理输出。

特点：

```text
高 HP
高防御
低速度
可保护后排
适合对抗物理敌人
```

#### 游侠

定位：后排物理输出、优先击杀脆弱目标。

特点：

```text
中等攻击
高命中
较高速度
适合打后排、飞行、小怪
```

#### 牧师

定位：治疗、解毒、净化、抗性辅助。

特点：

```text
低攻击
中等 HP
可治疗
可解除毒/诅咒
适合长线副本
```

#### 盗贼

定位：高闪避、破甲、偷袭、针对特殊目标。

特点：

```text
高速度
高闪避
可攻击后排
可破甲
适合对抗重甲或关键敌人
```

可选扩展职业：

#### 法师

定位：魔法输出、群体伤害、处理高物防敌人。

#### 守卫/骑士

定位：更专门的防御职业，强化保护、嘲讽、减伤。

MVP 若追求最小可行，可以先只做战士、游侠、牧师、盗贼四个职业。

## 7. 阵型系统

### 7.1 阵型结构

战斗使用 3×3 格子。

```text
前排左   前排中   前排右
中排左   中排中   中排右
后排左   后排中   后排右
```

每格最多 1 个单位。

玩家与敌人各自拥有一个 3×3 阵型。

### 7.2 阵型意义

阵型影响：

1. 被攻击概率。
2. 可攻击目标。
3. 技能范围。
4. 保护关系。
5. 后排安全性。
6. 范围技能命中目标。
7. 特定副本机制。

示例：

```text
前排更容易被近战攻击。
后排更安全，但可能被盗贼、弓手、法术或突袭技能针对。
中排适合辅助和半肉角色。
左右位置可以影响横扫、穿刺、溅射等技能。
```

### 7.3 MVP 阵型规则

MVP 不宜过度复杂。

建议先实现：

1. 近战优先攻击敌方前排。
2. 远程可以攻击中后排。
3. 保护技能可以替相邻或后方单位承受攻击。
4. 部分敌人拥有后排攻击能力。
5. 部分技能攻击一列或一排。

## 8. 战术配置系统

玩家不能在战斗中手操角色，而是在战前配置角色行为。

### 8.1 技能优先级

每个角色可以设置技能优先级。

示例：

```text
牧师：
1. 若任意队友 HP < 40%，使用治疗术
2. 若任意队友中毒，使用解毒
3. 否则使用普通攻击

战士：
1. 若后排受到威胁，使用保护
2. 若自身 HP < 30%，使用防御姿态
3. 否则攻击前排敌人

游侠：
1. 优先攻击敌方后排
2. 优先攻击低 HP 敌人
3. 否则攻击最近敌人
```

### 8.2 目标优先级

目标优先级可以包括：

```text
最近敌人
最低 HP 敌人
最高攻击敌人
敌方后排
敌方治疗单位
敌方精英单位
带有特定状态的敌人
```

MVP 不需要做复杂脚本语言，可以用下拉选项或规则槽实现。

### 8.3 使用条件

技能/道具使用条件包括：

```text
自身 HP 低于 X%
队友 HP 低于 X%
敌人数量大于等于 X
目标带有某状态
自身未处于某状态
技能剩余次数大于 0
当前层数大于等于 X
```

### 8.4 撤退条件

撤退条件非常重要，因为副本是多层结构，HP 和资源会在层间传递。

可配置撤退条件：

```text
任意角色 HP 低于 20%
两名以上角色 HP 低于 40%
牧师技能次数耗尽
消耗品耗尽
战士重伤
进入 Boss 层前队伍总 HP 低于 50%
```

MVP 中可以先提供预设撤退策略：

```text
保守：任意角色 HP < 35% 就撤退
标准：任意角色 HP < 20% 就撤退
激进：核心角色不倒下就继续
死战：不撤退
```

后续版本再开放更细的条件编辑。

## 9. 战斗系统

### 9.1 战斗方式

战斗为自动回合制。

基本流程：

```text
进入一层战斗
→ 根据速度决定行动顺序
→ 单位按照战术配置选择技能
→ 选择目标
→ 命中/闪避判定
→ 计算伤害和状态
→ 检查死亡、胜利、失败、撤退
→ 进入下一回合
```

### 9.2 行动顺序

MVP 推荐使用速度决定行动条或回合内排序。

简单版本：

```text
每回合开始时，所有存活单位按速度从高到低行动。
行动次数属性决定该单位每回合可行动几次。
```

更复杂的行动条系统可以后续再做。

### 9.3 伤害类型

MVP 伤害类型建议包含：

```text
物理
毒
魔法
诅咒
火焰
```

伤害可以同时拥有多个类型。

例如：

```text
毒刃 = 物理 + 毒
火球 = 魔法 + 火焰
诅咒箭 = 物理 + 诅咒
```

### 9.4 状态效果

MVP 状态效果建议控制在少量但有明确解法的类型：

```text
中毒：层间持续损失 HP
流血：战斗中持续损失 HP
诅咒：降低命中/治疗效果
眩晕：跳过下一次行动
破甲：降低防御
燃烧：持续火焰伤害，并增加装备耐久损耗
```

### 9.5 战斗结果

战斗结果分为：

```text
胜利：通过当前层，继续下一层或完成副本
失败：队伍被击溃，副本未完成
撤退：满足撤退条件后主动退出副本
```

失败和撤退的区别：

```text
失败：角色受到更严重伤病，损失更多资源
撤退：保留部分状态和已获得情报，但失去当天远征机会
```

MVP 中，副本失败或撤退不直接清除副本实例。

## 10. 副本系统

### 10.1 核心定义

副本是有生命周期的多层战术谜题，而不是单纯的一场战斗或一串随机怪物。

每个副本由两部分构成：

```text
副本模板：稳定设计内容
副本实例：本次刷新出来的具体变体
```

### 10.2 副本模板

副本模板决定：

```text
副本名
主题
推荐等级
核心机制
常见敌人
层数结构
基础奖励
常见掉落
解法方向
可能词缀池
可能事件池
Boss 或精英类型
```

模板的作用是保证副本可记忆。

玩家应该能逐渐形成认知：

```text
蜘蛛巢穴 = 毒 + 小怪 + 解毒/清杂
盗贼营地 = 后排威胁 + 闪避 + 命中
古代墓穴 = 诅咒 + 魔法抗性 + 持久战
熔岩裂谷 = 火焰持续伤害 + 装备耐久压力
废弃矿洞 = 高护甲 + 物理压力 + 矿石奖励
兽穴 = 流血 + 爆发 + 皮革/毒抗资源
```

### 10.3 副本实例

副本实例决定：

```text
本次出现的剩余天数
本次奖励次数
本次词缀
本次可见情报
本次敌人细节
本次特殊事件
本次奖励修正
本次隐藏条件
```

实例的作用是保证新鲜感。

同一个副本模板每次刷新时可以略有不同，但不能破坏副本主题。

例如：

```text
蜘蛛巢穴 + 毒雾弥漫
蜘蛛巢穴 + 精英盘踞
蜘蛛巢穴 + 宝藏暴露
蜘蛛巢穴 + 侦察困难
```

这些变体都仍然围绕“毒、小怪、持续消耗”展开。

### 10.4 副本生命周期

每个副本实例拥有生命周期。

MVP 推荐：

```text
普通副本：存在 3 天
精英副本：存在 2 天
特殊事件副本：存在 1 天
主线/最终副本：存在至期限结束
```

每个副本实例拥有奖励次数。

示例：

```text
普通副本：奖励通关 1 次
资源副本：奖励通关 2 次
精英副本：奖励通关 1 次
最终副本：奖励通关 1 次
```

奖励次数耗尽后，副本可以仍然存在，但再次通关只给少量经验或基础材料。MVP 也可以简单处理为奖励次数耗尽后副本消失。

### 10.5 多层结构

每个副本由 3~5 层组成。

每层应有明确功能：

```text
第 1 层：暴露副本主题
第 2 层：加入机制压力
第 3 层：制造配置考验
第 4 层：资源消耗或变化层
第 5 层：Boss 或精英验证
```

示例：蜘蛛巢穴

```text
第 1 层：小蜘蛛群，展示多小怪压力
第 2 层：毒蛛出现，展示中毒机制
第 3 层：蛛网陷阱，降低速度或闪避
第 4 层：精英毒蛛 + 小蜘蛛，考验清杂与解毒
第 5 层：蛛后 Boss，考验持续治疗、解毒和输出节奏
```

MVP 中不是所有副本都必须 5 层。早期副本可以 3 层，中后期副本使用 4~5 层。

## 11. 情报系统

### 11.1 情报分层

副本情报分为三层：

```text
公开情报
侦察情报
战后复盘情报
```

### 11.2 公开情报

公开情报在副本出现时直接显示。

包含：

```text
副本名
危险等级
预计层数
主要奖励
模糊威胁
剩余天数
奖励次数
可能伤害类型的模糊提示
```

示例：

```text
蜘蛛巢穴
危险等级：2
预计层数：4
主要奖励：皮革、毒囊、低级饰品
模糊威胁：大量小型敌人，存在持续性伤害
剩余天数：3
奖励次数：1
```

### 11.3 侦察情报

侦察后显示更具体信息。

包含：

```text
敌方阵型
主要敌人类型
主要伤害类型
抗性倾向
精英怪存在
陷阱类型
Boss 技能提示
特殊层机制
隐藏奖励条件
```

示例：

```text
侦察结果：
- 第 2 层开始出现毒蛛。
- 敌人主要造成物理 + 毒伤害。
- Boss 会周期性召唤小蜘蛛。
- 队伍若缺少解毒能力，可能在层间持续损失 HP。
- 火焰伤害对蜘蛛类敌人较有效。
```

### 11.4 战后复盘情报

战后复盘情报通过战报逐步补完。

包含：

```text
实际敌方技能
伤害统计
异常状态来源
命中/闪避统计
治疗与承伤统计
关键失败原因
隐藏机制揭示
```

示例：

```text
战后发现：
- 72% 的总伤害来自毒。
- 牧师在第 3 层前已经耗尽解毒次数。
- 后排游侠受到 4 次蛛网束缚，输出效率下降 38%。
- Boss 在小蜘蛛存活超过 2 回合时会获得攻击强化。
```

战后情报应帮助玩家理解失败，而不是只告诉玩家“你输了”。

## 12. 副本词缀系统

词缀用于给副本实例增加变化，但不能破坏副本模板主题。

MVP 词缀示例：

```text
精英盘踞：出现额外精英怪，奖励提高
毒雾弥漫：每层开始时全队有概率中毒
宝藏暴露：奖励提高，但 Boss 更强
时间紧迫：剩余天数更短，奖励更好
侦察困难：侦察只能获得部分情报
怪物暴动：敌方速度提高
重甲敌群：敌人防御提高，破甲收益提高
陷阱密布：层间事件更危险
资源丰富：材料掉落提高
Boss 变异：最终层 Boss 获得额外技能
```

词缀设计原则：

1. 改变解题侧重点。
2. 不改变副本基本身份。
3. 让玩家有机会通过情报或经验预判。
4. 奖励应与风险匹配。
5. 不产生无解组合。

## 13. 奖励与成长系统

### 13.1 奖励设计原则

奖励应与副本机制绑定。

玩家选择副本时，不应只是在选最高收益，而是在选择自己的成长路线。

示例：

```text
废弃矿洞：矿石、重甲、盾牌材料
蜘蛛巢穴：毒囊、皮革、毒抗饰品
盗贼营地：金币、轻甲、弓弩、命中装备
古代墓穴：法术材料、抗诅咒饰品、魔法装备
熔岩裂谷：火焰材料、耐久强化材料、火抗装备
兽穴：皮革、流血抗性资源、野性武器
```

### 13.2 成长线

MVP 包含三条主要成长线：

```text
角色成长：经验、等级、技能解锁
装备成长：掉落、购买、制作、替换
资源成长：金币、材料、消耗品、修理与训练
```

### 13.3 角色成长

角色通过远征获得经验。

升级可以带来：

```text
基础属性提升
技能槽增加
新技能解锁
技能次数提升
少量战术选项增加
```

MVP 中升级不宜太复杂。建议每个职业 3~5 个核心技能即可。

### 13.4 装备成长

装备分为：

```text
武器
护甲
饰品
消耗品
```

装备属性方向：

```text
攻击
防御
HP
命中
闪避
速度
特定抗性
技能次数
特殊效果
```

MVP 中装备词条应少而清晰，避免一开始做暗黑式复杂随机词条。

推荐采用：

```text
固定装备 + 少量品质差异
```

例如：

```text
铁剑：攻击 +5
猎弓：攻击 +3，命中 +8
厚重盾牌：防御 +6，速度 -2
毒抗戒指：毒抗 +15
圣徽：诅咒抗性 +10，治疗效果 +5%
```

### 13.5 制作与训练

MVP 中制作和训练发生在日终结算。

制作用于：

```text
制造装备
制造消耗品
修理装备
升级基础装备
```

训练用于：

```text
提升角色经验
解锁技能
恢复状态
```

MVP 不需要复杂生产链。只需要让玩家感到“今天不打这个副本，也可以为后续挑战做准备”。

## 14. 资源与限制系统

由于游戏取消现实时间等待，需要使用游戏内限制控制节奏。

### 14.1 核心限制

MVP 中建议包含：

```text
有限天数
每日远征次数
副本生命周期
奖励次数
角色 HP
异常状态
技能次数
消耗品数量
装备耐久
金币
材料
角色伤病
```

### 14.2 每日远征次数

每日默认 2 次远征。

远征可以用于：

```text
侦察副本
挑战副本
处理特殊事件
```

远征次数是玩家最重要的战略资源。

### 14.3 角色状态

角色在战斗中损失 HP，并可能获得异常状态。

副本层间会继承：

```text
HP 损失
中毒
诅咒
燃烧
技能次数消耗
消耗品消耗
装备耐久损耗
```

日终可以恢复一部分 HP，但重伤和异常状态需要治疗或休息。

### 14.4 装备耐久

装备耐久用于限制连续刷副本。

耐久降低会产生：

```text
属性下降
修理费用增加
装备损坏风险
```

MVP 中可以先简化为：

```text
每次副本后装备损失固定耐久。
特定副本和状态会额外消耗耐久。
耐久为 0 时装备失效，但不永久消失。
```

### 14.5 伤病

失败比撤退产生更严重后果。

示例：

```text
轻伤：次日 HP 上限 -10%
重伤：需要休息 1~2 天
濒死：需要治疗费用，数天无法出战
```

MVP 不建议使用永久死亡，除非游戏明确走硬核路线。

## 15. 战报系统

战报是本游戏的核心反馈系统。

战报必须让玩家知道：

1. 发生了什么。
2. 为什么赢。
3. 为什么输。
4. 哪些配置有效。
5. 下次可以怎么调整。

### 15.1 战报结构

每次远征生成一份战报。

战报包含：

```text
一、总结
二、每层结果
三、队伍状态变化
四、伤害统计
五、承伤统计
六、治疗统计
七、异常状态统计
八、技能使用统计
九、资源消耗
十、关键回合
十一、机制发现
十二、失败/撤退原因
```

### 15.2 总结示例

```text
远征结果：撤退
副本：蜘蛛巢穴
推进层数：第 3 层撤退
主要原因：毒伤累积过高，牧师解毒次数提前耗尽
获得：蜘蛛丝 ×3，毒囊 ×1，经验 42
损失：治疗药水 ×1，战士护甲耐久 -8，游侠中毒
新情报：蛛后会在小蜘蛛存活时获得攻击强化
```

### 15.3 数据统计示例

```text
总伤害来源：
毒伤害：52%
物理伤害：41%
其他：7%

我方输出：
游侠：46%
盗贼：31%
战士：18%
牧师：5%

我方承伤：
战士：61%
盗贼：19%
游侠：12%
牧师：8%

关键问题：
- 队伍缺少群体清杂。
- 解毒次数不足。
- 游侠在第 2 层后输出下降明显。
```

### 15.4 回合详情

MVP 可以提供折叠式回合详情。

默认展示摘要，玩家需要时再展开具体回合。

## 18. 前后端设计思路与技术路线

### 18.1 为什么 MVP 必须有前端

本游戏的核心体验不是单纯的数值模拟，而是围绕“信息展示、规划决策、战术配置、战报复盘”展开。

因此，MVP 不能只做命令行或后端规则模拟。命令行可以验证规则正确性，但很难验证以下关键问题：

```text id="u6ttoy"
玩家能否快速看懂副本列表？
玩家能否理解公开情报、侦察情报和战后情报的区别？
玩家能否在 UI 中自然完成站位、装备和技能配置？
玩家能否从战报中找到失败原因？
玩家能否理解当天远征次数、副本生命周期和奖励次数带来的取舍？
玩家是否会因为信息展示方式不同而做出不同决策？
```

所以 MVP 前端的定位不是最终美术界面，而是一个“玩法检验台”。

它需要优先服务以下目标：

```text id="rkn7vr"
清晰展示当前局势
降低配置队伍的操作成本
突出副本机制与风险
让战报可读、可复盘
快速暴露核心循环是否成立
```

MVP 前端可以很朴素，但不能缺失。
如果没有前端，就很容易把游戏做成“规则上成立，实际玩起来很难理解”的系统。

## 19. 总体技术路线

### 19.1 推荐路线

MVP 推荐采用：

```text id="l0ob28"
前端：React + TypeScript + Vite
后端：Python + FastAPI
存档：SQLite 或 JSON 文件
规则核心：独立 Python game_core 模块
通信方式：HTTP REST API
运行方式：本地启动后端 + 浏览器访问前端
```

该路线的目标是快速做出一个可玩的本地 Web 原型。

### 19.2 为什么不建议一开始做纯客户端

纯客户端也能做单机游戏，但本项目的规则系统较重，包括：

```text id="cljwpn"
副本刷新
多层副本推进
自动战斗
技能优先级
目标选择
战报生成
掉落结算
状态传递
存档读写
数据表校验
```

如果一开始全部写在前端，后续规则会和 UI 强耦合，不利于测试、复盘和扩展。

本项目更适合采用“规则核心独立，前端只负责展示和提交决策”的结构。

### 19.3 为什么不建议一开始做完整桌面客户端

MVP 阶段不建议一开始上 Electron、Tauri、Unity 或 Godot。

原因是当前最需要验证的是：

```text id="368lte"
核心循环是否成立
玩家是否理解情报
战术配置是否有意义
战报是否能支持复盘
副本生命周期是否制造取舍
```

这些问题用 Web 前端已经足够验证。

等核心玩法成立后，再考虑：

```text id="zrhqjy"
Tauri 桌面封装
Electron 桌面封装
Godot 重做表现层
Unity 重做表现层
移动端适配
```

MVP 不要让客户端形态绑架玩法验证。

## 20. 前后端职责划分

### 20.1 后端职责

后端负责所有游戏规则和权威状态。

后端应负责：

```text id="lbgmu5"
创建新游戏
读取存档
保存存档
生成每日世界刷新
生成副本实例
处理侦察
处理队伍配置
处理装备更换
处理技能配置
处理远征安排
执行自动战斗
结算奖励和损失
生成战报
推进到下一天
校验所有玩家操作是否合法
```

后端是游戏规则的唯一来源。

前端不能直接修改角色 HP、金币、装备、经验、副本状态等核心数据。
前端只能提交玩家意图，由后端校验并返回结果。

### 20.2 前端职责

前端负责展示、交互和体验验证。

前端应负责：

```text id="q6z9ey"
展示每日总览
展示副本列表
展示副本详情
展示公开情报、侦察情报和战后情报
展示队伍状态
提供 3×3 阵型配置界面
提供装备更换界面
提供技能优先级配置界面
提供目标优先级配置界面
提供撤退条件配置界面
提供远征顺序安排界面
展示结算过程
展示战报摘要和详情
提示玩家关键风险
```

前端不负责战斗计算，不负责掉落计算，不负责世界刷新计算。

### 20.3 规则核心职责

建议把游戏规则从 Web 后端中进一步拆出独立模块。

结构上可以分为：

```text id="oh02ar"
game_core：纯游戏规则
api_server：HTTP API
web_client：前端页面
```

其中 `game_core` 不依赖 FastAPI，也不依赖数据库。

`game_core` 只做纯规则计算，例如：

```text id="q05zvx"
apply_world_refresh(state)
resolve_scout(state, dungeon_id, scout_party)
resolve_expedition(state, expedition_plan)
resolve_battle(party, dungeon_layer, tactics)
apply_rewards(state, rewards)
generate_battle_report(battle_result)
advance_day(state)
```

这样做的好处是：

```text id="9g2tf5"
规则可以单元测试
战斗可以批量模拟
平衡可以自动跑样本
API 出问题时可以单独定位
未来可以替换前端但保留规则核心
未来也可以做 CLI 调试工具
```

## 22. 后端架构设计

### 22.1 后端分层

后端建议分为四层：

```text id="0elnn8"
API 层：接收前端请求，返回响应
Service 层：组织一次完整业务流程
Core 层：执行纯游戏规则
Persistence 层：负责存档读写
```

示例：

```text id="v1f8ke"
前端点击“结束当天”
→ POST /game/end-day
→ API 层接收请求
→ Service 层读取当前存档
→ Core 层按顺序结算远征
→ Core 层生成战报和新状态
→ Persistence 层保存状态
→ API 层返回结算结果
→ 前端展示战报和新一天状态
```

### 22.2 后端核心原则

后端应遵守以下原则：

```text id="sdxtqh"
所有规则计算都在后端进行
所有玩家操作都必须校验
所有随机结果都必须可复现
所有状态变化都应能追踪
所有战斗都应能生成战报
所有结算都应能被测试
```

### 22.3 随机种子

本游戏需要大量随机刷新，但必须支持复盘和调试。

每局游戏应有一个固定 `run_seed`。

每次随机事件可以派生子种子：

```text id="wp0ehm"
world_refresh_seed
dungeon_instance_seed
battle_seed
loot_seed
event_seed
```

这样可以实现：

```text id="ybb33s"
同一局游戏可复现
同一场战斗可复现
Bug 可以重放
平衡测试可以批量跑
战报可以追踪来源
```

### 22.4 状态存储

MVP 阶段可以优先使用 JSON 存档。

JSON 存档优点：

```text id="e7wgyr"
简单直观
方便调试
方便手动修改
方便查看完整游戏状态
适合单机 MVP
```

当系统稳定后，可以切换到 SQLite。

SQLite 适合：

```text id="4u2awh"
多存档
战报历史查询
长期统计
装备和副本历史记录
平衡分析
```

MVP 推荐路线：

```text id="sixg9f"
第一阶段：JSON 存档
第二阶段：SQLite 存档
第三阶段：导出/导入存档和战报
```

### 22.5 数据驱动

副本、敌人、技能、装备、职业、词缀都应尽量数据驱动。

也就是说，不要把具体内容硬编码到战斗逻辑里。

推荐使用数据表：

```text id="jrzkx4"
classes.json
skills.json
equipment.json
enemies.json
dungeon_templates.json
affixes.json
events.json
loot_tables.json
```

游戏逻辑负责解释数据，内容设计通过改数据完成。

这对 MVP 很重要，因为副本平衡一定会频繁调整。

## 23. API 设计草案

### 23.1 基础游戏接口

```text id="c2smu3"

创建新游戏

获取当前游戏状态

保存当前游戏

读取存档

结束当天并结算远征
```

```text id="6lr77l"

获取副本详情

安排侦察

POST /dungeons/{dungeon_id}/challenge
安排挑战
```

注意：
`scout` 和 `challenge` 在 MVP 中可以不是立即执行，而是加入当天远征计划。

```text id="bvhq0e"
获取队伍状态

更新阵型

更新战术配置

更新撤退条件
```

### 23.4 远征计划接口

```text id="wd54dq"
GET /expedition-plan
获取今日远征计划

更新今日远征计划
```

### 23.5 战报接口

```text id="yi7nvp"
获取回合详情
```

### 23.6 调试接口

MVP 开发阶段可以保留调试接口，但不要暴露给正式模式。

```text id="qfkmwc"

POST /debug/spawn-dungeon
```

## 24. 前端架构设计

### 24.1 前端核心页面

MVP 前端至少包含以下页面：

```text id="ygn21h"
每日总览页
副本列表页
副本详情页
队伍管理页
阵型配置页
战术配置页
远征计划页
战报页
商店/招募页
```

### 24.2 每日总览页

每日总览页是玩家进入一天后的主页面。

需要展示：

```text id="2cum2d"
当前天数
剩余天数
今日可用远征次数
金币与关键材料
队伍健康状态
当前副本数量
即将过期副本
今日推荐关注事项
结束当天按钮
```

它应该回答玩家的第一个问题：

> 今天我有什么机会，什么风险，什么事情快来不及了？

### 24.3 副本列表页

副本列表页应让玩家快速比较副本。

每个副本卡片展示：

```text id="ilymrv"
副本名
危险等级
剩余天数
奖励次数
预计层数
主要奖励
模糊威胁
是否已侦察
是否已挑战
词缀
```

需要突出：

```text id="em6iyb"
快过期副本
高奖励副本
已侦察副本
高风险副本
适合当前队伍的副本
```

### 24.4 副本详情页

副本详情页是玩家分析副本的核心界面。

应分为三块：

```text id="7o92yv"
公开情报
侦察情报
历史战报情报
```

副本详情页应该清楚显示：

```text id="a99x4l"
这个副本大概考验什么
我已经知道什么
我还不知道什么
侦察可能带来什么价值
挑战可能获得什么
失败或撤退会损失什么
```

### 24.5 阵型配置页

阵型配置页使用 3×3 格子。

需要支持：

```text id="qh7o5o"
拖拽角色到格子
点击角色查看属性
显示前/中/后排
显示左/中/右列
显示空格子
提示站位风险
```

示例风险提示：

```text id="f8dj40"
后排没有保护
牧师位于容易被突袭的位置
前排承伤角色 HP 偏低
当前阵型缺少中排支援
```

### 24.6 战术配置页

战术配置页负责配置自动战斗行为。

MVP 中不要一开始做复杂脚本编辑器。

推荐使用规则槽：

```text id="bsxx21"
技能优先级槽
目标优先级槽
道具使用槽
撤退条件槽
```

例如：

```text id="6egb7y"
牧师战术：
1. 队友 HP < 40% → 治疗术
2. 队友中毒 → 解毒
3. 否则 → 普通攻击

游侠战术：
1. 优先攻击后排
2. 优先攻击低 HP
3. 否则攻击前排
```

前端应尽量用自然语言展示规则，避免让玩家觉得像在写代码。

### 24.7 远征计划页

远征计划页用于安排当天行动。

展示：

```text id="byxqhv"
今日剩余远征次数
已安排行动
行动顺序
每次行动的队伍
目标副本
预估风险
可获得奖励
```

玩家可以：

```text id="q7lvf7"
添加侦察
添加挑战
移除行动
调整行动顺序
查看风险摘要
确认结束当天
```

### 24.8 战报页

战报页是核心反馈页面。

默认应先展示摘要，不要一上来堆满回合日志。

推荐结构：

```text id="msf4pz"
结果摘要
关键原因
每层概览
奖励与损失
数据统计
机制发现
建议复盘点
回合详情
```

战报页最重要的目标是让玩家知道：

```text id="6xai57"
为什么赢
为什么输
哪里出了问题
下次应该调整什么
```

### 24.9 商店与招募页

MVP 商店和招募可以简单。

商店展示：

```text id="a1fk4y"
装备
消耗品
材料
修理服务
```

招募展示：

```text id="vnmbyu"
候选角色
职业
等级
特长
招募费用
```

不需要复杂经济系统，只要能支持阵容扩展和路线选择。

## 25. 前端状态设计

### 25.1 前端状态原则

前端只保存 UI 状态和后端返回的视图状态，不保存权威游戏状态。

前端可以保存：

```text id="rnz9p1"
当前打开的页面
当前选中的副本
当前选中的角色
当前编辑中的阵型草稿
当前编辑中的战术草稿
当前展开的战报层级
加载状态
错误提示
```

前端不应该自行计算：

```text id="dyufmy"
战斗结果
掉落结果
副本刷新
角色升级
装备耐久损耗
异常状态结算
经验结算
```

这些都由后端完成。

### 25.2 前端 Store

前端可以有一个轻量 store。

示例：

```text id="hpnb1c"
gameState
selectedDungeonId
selectedCharacterId
expeditionPlanDraft
formationDraft
tacticsDraft
latestReportId
uiWarnings
```

### 25.3 乐观更新

MVP 阶段不建议大量使用乐观更新。

比如装备更换、阵型调整可以本地先显示草稿，但最终必须以后端确认结果为准。

推荐策略：

```text id="vt6b7c"
配置类操作：允许前端草稿
结算类操作：必须等待后端返回
存档类操作：以后端为准
战斗类操作：以后端为准
```

## 26. 前后端数据流

### 26.1 查看副本数据流

```text id="jjpwx9"
玩家打开副本列表
→ 前端 GET /dungeons
→ 后端读取当前 GameState
→ 后端生成 DungeonListView
→ 前端渲染副本卡片
```

### 26.2 侦察数据流

```text id="1fsj69"
玩家选择侦察某副本
→ 前端 POST /expedition-plan
→ 后端校验远征次数、角色可用性、副本是否存在
→ 后端更新今日远征计划
→ 前端刷新远征计划
```

结束当天后：

```text id="6kd1gu"
后端结算侦察
→ 解锁副本侦察情报
→ 生成侦察结果
→ 写入 GameState
→ 前端展示新情报
```

### 26.3 挑战副本数据流

```text id="lovpcu"
玩家安排挑战副本
→ 前端提交远征计划
→ 玩家点击结束当天
→ 后端读取计划
→ 后端调用 game_core.resolve_expedition
→ 后端逐层推进副本
→ 后端生成 BattleReport
→ 后端写入奖励、损失和新状态
→ 前端展示战报
→ 前端展示更新后的第 N+1 天状态
```

### 26.4 配置队伍数据流

```text id="f9ovbk"
玩家调整阵型/装备/技能
→ 前端生成配置草稿
→ 玩家保存配置
→ 前端 POST /party/formation 或 /party/tactics
→ 后端校验配置合法性
→ 后端保存到 GameState
→ 前端刷新队伍状态
```

## 27. 前端 MVP 原型优先级

### 第一优先级：能玩完整一天

必须完成：

```text id="sphmqu"
每日总览页
副本列表页
副本详情页
队伍状态展示
远征计划页
结束当天按钮
结算结果展示
```

达到标准：

> 玩家可以从第 1 天进入，查看副本，安排远征，结束当天，看到结果，进入第 2 天。

### 第二优先级：能配置队伍

必须完成：

```text id="xuj013"
阵型配置
装备更换
基础技能优先级
基础撤退策略
```

达到标准：

> 玩家可以因为某个副本机制调整站位和装备，并在战报中看到差异。

### 第三优先级：能复盘失败

必须完成：

```text id="rayjg4"
战报摘要
层数结果
伤害统计
承伤统计
状态统计
关键失败原因
机制发现
```

达到标准：

> 玩家失败后能知道主要问题是毒伤、后排被打、命中不足、治疗耗尽，还是撤退条件太激进。

### 第四优先级：能支撑成长路线

必须完成：

```text id="7g60zw"
商店
招募
装备掉落
角色升级
材料获取
简单制作或修理
```

达到标准：

> 玩家会为了打某个副本而先去刷材料、买装备、招募角色或调整成长方向。


达到标准：

> 同一个副本，用不同站位、装备和技能配置，会产生明显不同的结果。

### 第四优先级：副本与情报

必须完成：

```text id="a39xz2"

公开情报
侦察情报
战后情报
副本生命周期
副本词缀
奖励次数
```

达到标准：

> 副本不是一次性战斗，而是有情报、有生命周期、有多层压力的战术谜题。

### 第五优先级：战报生成

必须完成：

```text id="eer6ko"
战斗摘要
每层结果
伤害统计
治疗统计
承伤统计
状态统计
技能使用统计
关键事件
失败原因
机制发现
```

达到标准：

> 战报不仅记录过程，还能帮助玩家做下一次决策。

## 29. 本地开发与调试方案

### 29.1 本地运行方式

MVP 本地开发可以采用两个进程：

```text id="i8v4pl"
后端：localhost:8000
前端：localhost:5173
```

前端通过 API 请求后端。

### 29.2 调试面板

MVP 前端建议内置一个开发者调试面板。

调试面板可以提供：

```text id="t0dd93"
查看完整 GameState
查看当前随机种子
强制刷新副本
强制生成指定副本
给角色添加经验

给玩家添加装备
直接跳到指定天数
重放上一场战斗
导出战报 JSON
导入测试存档
```

这会极大提高平衡调试效率。

### 29.3 战斗模拟工具

后端应提供一个简单模拟工具，用于批量测试。

示例用途：

```text id="94n918"

不同副本危险等级的通关率

某个职业是否过弱
某个词缀是否制造无解组合
```

即使正式游戏是单机确定性，也需要模拟工具辅助平衡。

### 29.4 日志与可复现

每次结算应记录：

```text id="hzwtib"

```

这样当出现“我不知道为什么输了”或“这个副本是不是数值坏了”时，可以复盘。

## 31. 前后端协作原则

### 31.1 前端驱动体验，后端保证规则

前端负责让玩家理解局势，后端负责保证规则正确。

不要让前端直接计算关键结果。

### 31.2 API 应围绕玩家动作设计

接口不应该只围绕数据库 CRUD 设计，而应该围绕玩家动作设计。

好的接口：

```text id="sut850"
安排侦察
安排挑战
保存阵型
保存战术
结束当天
查看战报
```

较差的接口：

```text id="p122gf"
修改角色表
修改副本表
修改装备表
修改状态字段
```

玩家动作接口更接近游戏设计，也更容易维护规则合法性。

### 31.3 战报是核心产物，不是附属日志

战报不应该只是 debug log。

战报应该是游戏体验的一部分，需要作为正式数据模型设计。

### 31.4 前端优先暴露体验问题

如果玩家觉得难理解，优先怀疑：

```text id="tq927o"
情报展示不清楚
副本卡片信息层级不对
战报没有指出关键原因
队伍配置入口太深
风险提示不够明确
奖励和副本机制关联不明显
```

不要第一时间只改数值。

## 32. 技术路线总结

MVP 推荐路线：

```text id="rjdv4g"
后端用 Python 实现核心规则
FastAPI 提供本地 HTTP API
前端用 React + TypeScript + Vite
存档先用 JSON，稳定后切 SQLite
规则核心独立成 game_core
所有内容尽量数据驱动
所有战斗可复现
所有战报可追踪
前端作为玩法检验台
```

这条路线的核心优势是：

```text id="r7bpcn"
开发速度快
规则容易测试
前端容易迭代
战斗方便调试
数据方便调整
未来容易扩展
```

MVP 的最终目标不是做一个架构复杂的工程，而是用最短路径做出一个可以验证核心体验的本地可玩原型。

## 34. MVP 平衡原则

### 34.1 副本难度

副本难度不应只来自数值压制，而应来自机制问题。

好的失败原因：

```text
缺少解毒
缺少命中
后排保护不足
撤退条件不合理
治疗次数耗尽
装备耐久不足
没有针对敌方抗性
```

差的失败原因：

```text
敌人单纯攻击太高
敌人血量过厚
玩家完全看不懂为什么输
副本没有提供任何可利用情报
```

### 34.2 奖励节奏

MVP 中，每 2~3 天应给玩家一次明显成长。

成长形式可以是：

```text
新装备
新技能
新角色
新副本情报
新制作配方
新阵容思路
```

### 34.3 挑战节奏

玩家应该经常遇到暂时打不过的副本，但不应频繁遇到完全无解的副本。

理想状态：

```text
低风险副本：稳定获取资源
中风险副本：需要基本针对
高风险副本：需要侦察和准备
精英副本：可能需要专门构筑
最终副本：检验整局成长路线
```

## 36. MVP 验收标准

MVP 可以通过以下问题验收：

1. 玩家是否需要根据副本机制调整阵容？
2. 玩家是否会主动侦察高风险副本？
3. 玩家是否能从战报中理解失败原因？
4. 玩家是否会为了某个副本去刷特定装备或材料？
5. 玩家是否会在“硬打、侦察、准备、放弃”之间产生真实取舍？
6. 玩家是否能记住不同副本的战术特征？
7. 同一副本模板的不同实例是否有新鲜感？
8. 30 天内是否形成了清晰的成长节奏？
9. 自动战斗是否让玩家觉得自己的配置被验证了？
10. 游戏是否避免了无脑连续刷副本？

如果这些问题大多成立，则 MVP 方向正确。

## 38. 总结

本 MVP 的核心不是做一个内容量巨大的 RPG，而是做一个紧凑的战术经营循环。

它的关键结构是：

```text
有限天数制造战略压力
每日远征次数制造机会成本
副本生命周期制造取舍
情报分层制造规划空间
多层副本制造资源压力
自动战斗验证战术配置
战报系统支持复盘学习
奖励绑定副本机制与成长路线
```

最终目标是让玩家不断经历：

```text
看见机会
→ 判断风险
→ 获取情报
→ 调整构筑
→ 挑战失败
→ 阅读战报
→ 找到问题
→ 准备资源
→ 再次挑战
→ 成功攻克
→ 解锁新路线
```

这就是本游戏 MVP 最需要验证的核心体验。
